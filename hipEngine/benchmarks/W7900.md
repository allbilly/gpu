# W7900 benchmark sweep

Date: 2026-05-25

Diagnostic W7900 rerun of the hipEngine families used for the README comparison: hipEngine PARO and hipEngine GGUF, both under a clean TheRock ROCm 7.13 environment. Existing W7900 llama.cpp HIP/Vulkan Q4_K_M rows are reused because those baselines have not changed. Values are `prefill tok/s / decode tok/s / peak GiB`. hipEngine rows use one resident session per model allocated for the largest requested shape (`128K/128`) with 1 warmup + 5 measured runs per shape, so hipEngine peak is the max-context tracked allocator high-water mark; llama.cpp peak is sampled whole-card VRAM from existing per-shape artifacts.

These are diagnostic hardware rows, not promoted fastest rows: the refresh has repeated final-token sanity but no new KL/top-1 gate for the production shisa/PARO and GGUF rows, and llama.cpp rows are cross-quant Q4_K_M comparisons.

## Command templates

hipEngine GGUF Q4_K_S (single model load, 5 measured runs per shape):

```bash
CONDA_PREFIX=/home/lhl/mambaforge/envs/therock
ROOT=$($CONDA_PREFIX/bin/python3.12 -m rocm_sdk path --root)
env -i HOME=$HOME USER=$USER LOGNAME=$LOGNAME SHELL=$SHELL TERM=${TERM:-xterm} \
  PATH="$ROOT/bin:$CONDA_PREFIX/bin:/usr/local/bin:/usr/bin:/bin" \
  LD_LIBRARY_PATH="$ROOT/lib:$CONDA_PREFIX/lib/python3.12/site-packages/_rocm_sdk_core/lib:$CONDA_PREFIX/lib/python3.12/site-packages/_rocm_sdk_libraries_gfx110X_all/lib" \
  HIP_PATH="$ROOT" ROCM_PATH="$ROOT" HIP_LIB_PATH="$ROOT/lib" HIP_INCLUDE_PATH="$ROOT/include" \
  HSA_OVERRIDE_GFX_VERSION=11.0.0 PYTHONPATH=. \
  HIPENGINE_GGUF_DECODE_REPACK=1 \
  python scripts/qwen35_readme_sweep.py --engine gguf \
    --model /models/gguf/Qwen3.6-35B-A3B-UD-Q4_K_S.gguf --quant gguf_q4_k_s \
    --workloads 512/128 4K/128 32K/128 128K/128 \
    --warmup-runs 1 --measured-runs 5 --warmup-decode-tokens 1 \
    --force-bulk-prefill --bulk-prefill-attention-mode bulk \
    --use-wmma-prefill --use-gemv-decode \
    --compiler-version-file /tmp/hipengine-hipcc-version-713.txt --require-cached-build \
    --json /tmp/w7900-therock713-sweep/q4ks-readme-persistent-5run.json
```

## Interpretation highlights

- Loading once and running 5 measured repetitions removes the GGUF per-shape load/repack penalty from timing variance: Q4_K_S reaches `2577 tok/s` at 4K/128 and `1894 tok/s` at 32K/128 on W7900.
- hipEngine PARO remains the fastest prefill path at every shape in this W7900 sweep. The single max-context session means short-shape hipEngine memory is intentionally not a minimum-fit number; it is the allocator high-water mark for the full `128K/128` sweep session.
- The PARO 512/128 row benefits from the workspace-overlap fix and from measuring repeated runs after the one-time resident load/build. A follow-up W7900 threshold sweep found the overlap-minimizing path regressed 4K/8K/16K, was tied at 32K, and became non-negative/positive at 48K+ while saving ~0.2 GiB; current code therefore keeps workspaces resident through 32K while retaining overlap minimization for >32K long-context rows.
- llama.cpp Vulkan still owns the short-context decode column, while hipEngine PARO/GGUF are competitive or faster on prefill in the same broad model family. Cross-engine GGUF comparisons are cross-quant (`Q4_K_S` vs `Q4_K_M`) and should be read as approximate.
- TheRock `hipMemGetInfo` reports invalid low/negative values on this mixed setup, so hipEngine memory columns use tracked allocator peak rather than sampled HIP used memory.
