# RX 7900 XTX benchmark sweep

Date: 2026-05-23

One-shot local rerun of the benchmark families currently used for hipEngine comparisons: hipEngine PARO, hipEngine GGUF, llama.cpp HIP, and llama.cpp Vulkan. Values are `prefill tok/s / decode tok/s / peak GiB`. hipEngine peak is the tracked allocator high-water mark; llama.cpp peak is whole-card amdgpu VRAM sampled at 10 ms. These are diagnostic hardware rows, not promoted README fastest rows, because the run used single measurements and only harness finite-logit sanity rather than the full KL/top-1 correctness gate.

## Card settings and tuning notes

This RX 7900 XTX is intentionally power-limited because higher-power settings
previously caused host shutdowns. The settings below are a read-only capture from
2026-05-23; no GPU settings were changed while collecting them.

| Setting | Captured value | Notes |
| --- | --- | --- |
| Power cap | `290 W` | Below default `303 W`; sysfs min/default/max are `272/303/402 W`. |
| Performance level | `manual` | `power_dpm_force_performance_level=manual`. |
| Active power profile | `BOOTUP_DEFAULT*` | `COMPUTE` profile is available as profile index `5`, but was not active for these rows. |
| OD SCLK | `500 / 2500 MHz` | `pp_dpm_sclk` exposed idle/low/high states of about `22/500/2371 MHz` at capture. |
| OD MCLK | `97 / 1250 MHz` | `pp_dpm_mclk` states were `96/456/772/1249 MHz`. |
| Undervolt | `OD_VDDGFX_OFFSET=-50 mV` | Valid range reported by sysfs was `-450..0 mV`; this mild undervolt is part of the local stability/power setup. |
| PCIe link | sysfs current/max `16.0 GT/s x16`; rocm-smi idle DPM showed `8.0 GT/s x8` | The DPM view can downshift while idle; verify under load before blaming PCIe. |
| Idle thermals | edge/junction/mem about `33/41/46 C` | Fan was stopped/auto at idle. |

Potential higher-performance experiments, if the PSU/platform is stable enough:

1. **Measure clocks under load first.** The suspicious rows are more likely to be
   clock-ramp / power-profile artifacts than raw bandwidth limits. Log clocks,
   power, utilization, and thermals while a benchmark is running:

   ```bash
   while true; do
     date -Is
     rocm-smi -P -S -s -M -m -u -t
     sleep 0.2
   done | tee /tmp/rx7900xtx-bench-clocks.log
   ```

2. **Try compute profile without raising the power cap.** This may reduce clock
   ramp latency for short prompts while keeping the `290 W` cap:

   ```bash
   # inspect available profiles first
   cat /sys/class/drm/card1/device/pp_power_profile_mode

   # example only: set COMPUTE profile, then re-run 512/128 and 4K/128
   # use --setprofile 5 if this rocm-smi build does not accept the name
   sudo rocm-smi --setprofile COMPUTE
   rocm-smi --showprofile --showperflevel --showclocks --showpower
   ```

3. **Try deterministic high DPM states at the existing cap.** This is useful for
   separating clock-ramp effects from power-limit effects:

   ```bash
   # examples only; confirm indices on this boot with rocm-smi -S -s first
   sudo rocm-smi --setperflevel manual --setsclk 2 --setmclk 3
   rocm-smi -S -s -M -m -P
   ```

4. **Only then consider a higher cap.** The default cap is `303 W` and the driver
   reports max `402 W`, but this machine has already shown shutdown behavior at
   higher power. If testing, step gradually and monitor wall power and logs:

   ```bash
   # example only; do not use unless the shutdown issue is understood
   sudo rocm-smi --setpoweroverdrive 303
   ```

After any tuning run, restore the known-safe local setup before leaving the
machine unattended:

```bash
sudo rocm-smi --setpoweroverdrive 290
# Re-apply BOOTUP_DEFAULT/profile and local OD/undervolt settings as needed;
# verify with rocm-smi -P -S -s -M -m --showprofile --showperflevel.
```

## Interpretation highlights

- On this 24 GiB card, hipEngine PARO is the only hipEngine path in this report that reaches 128K/128; its BF16 KV 128K row is `1021.7 / 62.14 / 22.17`, and INT8 KV reduces tracked peak to `21.18 GiB`.
- hipEngine GGUF Q4_K_M is competitive at 4K/128 (`2425.4 / 99.23`) but memory-constrained beyond 32K/128 on the current resident allocation strategy.
- llama.cpp Vulkan has the best decode column for GGUF at every f16-KV row in this run; llama.cpp HIP has stronger 64K prefill than Vulkan in the f16-KV split run, while Vulkan recovers with Q8_0 KV.
